# Course Syllabus: Apache Kafka Fundamentals (Coursera)

> **Kafka 4.x annotations** are marked with `[Kafka 4.x]` throughout.
> Installed version: **Apache Kafka 4.3.1** (KRaft mode, ZooKeeper removed).

## 📋 Course Overview
This course covers the core mechanics of Apache Kafka, distributed messaging concepts, and building scalable real-time data pipelines.

* **Total Modules:** 7
* **Level:** Beginner
* **Estimated Time:** ~5 hours

## 📖 Module Breakdown

### 🔹 Module 1: Big Data and Messaging Fundamentals
* **Focus:** Understanding why real-time processing is required in a data-driven world.
* **Key Topics:**
  * The growth of modern data systems.
  * Significance of Big Data and real-time analytics applications.
  * Introduction to core messaging platforms and architectures.

> **Demo:** `module1/BigDataMessagingDemo.java` — no Kafka connection needed; structured console output.

---

### 🔹 Module 2: Introduction to Apache Kafka
* **Focus:** Deep dive into Kafka's unique architectural approach to distributed systems.
* **Key Topics:**
  * Apache Kafka features, core design, and industry use cases.
  * Core architecture concepts: Brokers, Topics, Partitions, and Replication.
  * How Kafka handles distributed messaging safely at scale.

> **[Kafka 4.x]** The cluster metadata layer is now managed by **KRaft** (Kafka Raft), not ZooKeeper.
> The **Cluster ID** is a UUID generated by `kafka-storage.sh` instead of being assigned by ZooKeeper.
> Controller election is handled by the KRaft quorum; `kafka-metadata-quorum.sh` replaces ZooKeeper-based tools.

> **Demo:** `module2/KafkaArchitectureDemo.java` — uses `AdminClient` to show live cluster ID, controller, brokers, and topics.

---

### 🔹 Module 3: Kafka Setup and Cluster Configuration
* **Focus:** Practical environment setup and basic cluster operations.
* **Key Topics:**
  * Setting up a local Kafka environment.
  * ~~Installing and configuring ZooKeeper.~~ *(removed in Kafka 4.0)*
  * Installing and spinning up Apache Kafka brokers.

> **[Kafka 4.x] ZooKeeper is GONE.** Kafka 4.0 removed ZooKeeper entirely. The setup sequence is now:
> ```bash
> # 1. Generate cluster UUID
> bin/kafka-storage.sh random-uuid
>
> # 2. Format the storage directory with KRaft metadata
> bin/kafka-storage.sh format -t <UUID> -c config/kraft/server.properties
>
> # 3. Start the broker (no separate ZooKeeper process)
> bin/kafka-server-start.sh config/kraft/server.properties
> ```
>
> **[Kafka 4.x] CLI flag change:** `--zookeeper` has been removed from all Kafka shell scripts.
> Use `--bootstrap-server localhost:9092` everywhere.
>
> **[Kafka 4.x] New tool:** `bin/kafka-metadata-quorum.sh --bootstrap-server localhost:9092 describe --status`
> replaces ZooKeeper-based cluster inspection.

> **Demo:** `module3/ClusterSetupDemo.java` — creates topic `kafka-demo` via `AdminClient`, shows partition/leader/ISR layout.

---

### 🔹 Module 4: Kafka Producer: Architecture and Configuration
* **Focus:** Learning how data enters a Kafka cluster.
* **Key Topics:**
  * Kafka Producer internals and execution flow.
  * Producer configuration parameters for tuning reliability and performance.
  * Basic producer terminal/CLI operations.

> **[Kafka 4.x] `enable.idempotence` is now `true` by default.**
> Previously the default was `false`; in 4.x an idempotent producer is the baseline.
> This means `acks` is also implicitly `all` when idempotence is on.
>
> **[Kafka 4.x]** `max.in.flight.requests.per.connection=5` is safe with idempotence (was 1 in older guides).

> **Demo:** `module4/ProducerArchitectureDemo.java` — fire-and-forget (acks=0), synchronous (acks=1), and async callback (acks=all + idempotent) patterns.

---

### 🔹 Module 5: Kafka Producer: Serialization and Partitioning
* **Focus:** How messages are structured and routed across brokers.
* **Key Topics:**
  * Working with Serializers to convert objects into stream bytes.
  * Understanding Partitioners and routing strategies (e.g., using message keys vs. round-robin).

> **[Kafka 4.x]** The default partitioner is the **Sticky Partitioner** (introduced in 2.4, now firmly the default).
> It fills one batch before switching partitions, improving throughput over pure round-robin.
> Key-based partitioning (murmur2 hash) is unchanged.
>
> **[Kafka 4.x]** The `Partitioner` interface is deprecated in favour of `ProducerInterceptor`-based approaches
> or configuring `partitioner.ignore.keys=true` for sticky behaviour on keyed records.

> **Demo:** `module5/SerializationPartitioningDemo.java` — custom `OrderEventSerializer` + custom `RegionPartitioner` (APAC/EMEA/AMER key routing).

---

### 🔹 Module 6: Kafka Consumer: Architecture and Configuration
* **Focus:** Learning how data is pulled out of Kafka topics.
* **Key Topics:**
  * Kafka Consumer internal architecture.
  * Reading messages using the CLI and client configurations.
  * Understanding Consumer Groups and the rebalancing mechanism.

> **[Kafka 4.x] KIP-848 — New Consumer Group Protocol.**
> Kafka 4.x ships the server-side incremental cooperative rebalance as stable:
> ```
> group.protocol=CONSUMER   # new default — incremental, no stop-the-world
> group.protocol=CLASSIC    # legacy eager rebalance (still supported)
> ```
> The new protocol eliminates the "stop-the-world" rebalance; only partitions
> that actually move are paused. This significantly reduces latency spikes on
> group membership changes.
>
> **[Kafka 4.x]** `ConsumerRebalanceListener` continues to work with the new protocol.
> New callback: `onPartitionsLost()` (added in 2.7) handles cases where the
> coordinator forcibly removes partitions without calling `onPartitionsRevoked`.

> **Demo:** `module6/ConsumerArchitectureDemo.java` — consumer group with `LoggingRebalanceListener`, reads up to 10 records, lists groups via `AdminClient`.

---

### 🔹 Module 7: Kafka Consumer: Offset Management and Deserialization
* **Focus:** Data evolution and message consumption tracking.
* **Key Topics:**
  * Using Deserializers to process incoming streams back into data structures.
  * Managing offsets to handle state, deduplication, and failure recovery.

> **[Kafka 4.x]** `enable.auto.commit=true` remains the default. For production
> at-least-once guarantees, disable it and use `commitSync()` or `commitAsync()`.
>
> **[Kafka 4.x] `DeserializationExceptionHandler`** (available via Kafka Streams;
> for plain consumers, handle `SerializationException` in a try/catch around
> `poll()` processing) allows routing poison-pill records to a dead-letter topic
> instead of crashing the consumer loop.
>
> **[Kafka 4.x]** Transactional consumers (`isolation.level=read_committed`) work
> with the new consumer group protocol for end-to-end exactly-once semantics.

> **Demo:** `module7/OffsetManagementDemo.java` — manual sync commit per batch, per-record commit with `seekToBeginning` replay, delivery semantics explained.

## GOAL — Implementation Notes

**[Kafka 4.x] Summary of breaking / notable changes affecting this course:**

| Syllabus Area | Old (3.x) | New (4.x) |
|--------------|-----------|-----------|
| Cluster metadata | ZooKeeper required | KRaft only; ZooKeeper removed |
| Setup | Start ZooKeeper first, then broker | Single `kafka-server-start.sh` |
| CLI flags | `--zookeeper localhost:2181` | `--bootstrap-server localhost:9092` |
| Java minimum | 11 | **17** (demo uses 21) |
| Idempotence default | `false` | **`true`** |
| Consumer rebalance | Eager (stop-the-world) | Incremental cooperative (KIP-848) |
| Partitioner | RoundRobin / Sticky | Sticky (default); `Partitioner` API deprecated |
| Metadata inspection | `zookeeper-shell.sh` | `kafka-metadata-quorum.sh` |
